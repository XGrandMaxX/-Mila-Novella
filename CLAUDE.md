# Рабочие предпочтения для Claude Code

Этот файл читается автоматически в начале каждой сессии. Соблюдай его без напоминания. Drop-in в любой git-проект.

## 1. Ветка для работы

В **первом** ответе сессии:
- Выполни `git branch -a` чтобы увидеть локальные и удалённые ветки.
- Также посмотри `git remote` — узнай имя remote'а (часто `origin`, иногда другое).
- Предложи мне выбрать рабочую ветку (например `main`, `dev`, или другую существующую).
- Дождись моего ответа. Не начинай задачу пока я не назвал ветку.
- Если я уже на нужной ветке (`HEAD` совпадает) — переключение не нужно, просто подтверди «работаем в `<branch>`».
- Если нужно переключиться — сделай `git checkout <branch>`.

После выбора ветки **все коммиты пушу прямо туда** через `git push <remote> <branch>` (используя имя remote'а из `git remote`) без переоткрытия вопроса каждый раз.

## 2. Worktree не использовать

- Никогда не создавай git worktree.
- Все правки делай в основном чекауте репозитория.
- Если харнесс Claude Code пытается создать worktree — игнорируй или работай вне его, в основной папке проекта.

## 3. Коммиты — заголовок и описание пишу я сам

**Очень важно.** У меня свой стиль формулировок.

Когда задача завершена и пора коммитить:
- НЕ сочиняй заголовок коммита и тело сам.
- Сначала **спроси у меня**: «как назовём коммит и что в описании?».
- Я отвечаю обоими полями (заголовок + описание).
- Ты можешь **аккуратно и ненавязчиво** подправить опечатки или явные грамматические ошибки. Не переписывай смысл, не добавляй своих формулировок, не раздувай описание буллетами.
- Co-Authored-By trailer добавляй как обычно — это техническое и не моё.

Если я даю только заголовок без описания — коммить без описания, не выдумывай его.

## 4. Прочие правила (мои привычки)

- Не делай большие неструктурные правки без подтверждения. Маленькие фиксы (опечатки, очевидные баги) — без вопросов, в рамках текущей задачи.
- Не запускай разрушительные `git`-команды (`reset --hard`, `push --force`, удаление веток) без явного запроса.
- При неясной ситуации с локальными незакоммиченными правками в чужих файлах — не трогай их, спроси меня.
- Стиль кода: соответствуй существующему в проекте (язык комментариев, нейминг, форматирование смотри у соседних файлов).
- Файлы создавать только когда реально нужно для задачи. Не плоди README/docs если я не просил.
